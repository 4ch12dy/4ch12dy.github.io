深入理解CoreLocation.framework定位原理

经过一周的逆向调试，终于把整个框架获取定位的来龙去脉梳理完成。下面主要从程序员开发的角度和底层实现两个去讨论这个问题

写在前面

从底层的实现来说，与之前我所理解的有所不同，之前认为是框架最终和内核交互获取定位信息；但后面发现，框架实际上并没有直接去从内核获取定位信息，而是通过XPC（苹果中进程通信框架）发送和接受消息去获取定位。而与其交互的是一个root权限的/usr/libexec/locationddeamon进程，这个守护进程负责与内核交互获取定位信息。所以如果说要去完全防止hook（框架中实际上很多api都能被hook拿到定位），可以去逆向他们之间的XPC交互协议，并通过这个协议应用去和locationd通信获取定位信息，下面就讨论我目前的研究发现。

在这之中，还有两种特殊的消息，分别是kCLConnectionMessagePausesLocationUpdatesAutomatically和kCLConnectionMessageWatchdog涉及到超时重新连接相关。

获取定位时XPC通信过程

分别从发送消息和接收消息来对应

  • 当发送一条kCLConnectionMessageRegistration注册消息时会收到同样的kCLConnectionMessageRegistration消息回复。
  • 当发送一条kCLConnectionMessageRequestAuthorization授权请求消息时会收到kCLConnectionMessageAuthorizationStatus消息回复，暗示授权的状态
  • 当发送一条kCLConnectionMessageLocation定位消息的时候会收到kCLConnectionMessageLocation消息
  • 当收到一条xpc_error_connection_interrupted消息的时候，会发送kCLConnectionMessageWatchdog消息然后重复之前的步骤。
